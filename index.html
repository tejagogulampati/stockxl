Must-Know Definitions

Every key term — clear, short, memorizable.

core concept
Process
A program that is currently running in memory. A program is a file on disk; a process is what it becomes when it starts executing. Multiple processes can exist from the same program.
Opening Chrome twice = two separate processes, both from the same Chrome program.
identifier
PID — Process ID
A unique number the OS assigns to every process so it can track and manage it. Like a student ID number.
Linux command ps -ax lists every process with its PID.
data structure
PCB — Process Control Block
A data record the OS keeps for every process containing everything needed to manage it: PID, name, child process list, accounting info (run time, cost), open file list, CPU registers, instruction pointer, memory pointers, message queues.
Think of it as the process's "folder" at the OS office — everything about it in one place.
lifecycle
Process States
The five stages a process moves through: New (being loaded), Ready (waiting for CPU), Running (on the CPU now), Waiting/Blocked (waiting for I/O or event), Exit (finished, being cleaned up).
OS component
Scheduler (Process Scheduler)
The part of the OS that decides which process runs, when, and for how long. It uses a scheduling algorithm (policy) to make these decisions.
metrics
Wait Time, Turnaround Time, Throughput
Wait Time — How long a process spent waiting (not running) from arrival to completion.
Turnaround Time — Total time from arrival to completion (wait + run).
Throughput — How many processes complete per unit of time.
Formulas: Turnaround = Completion − Arrival  |  Wait = Turnaround − Run Time
scheduling type
Non-Preemptive vs. Preemptive
Non-Preemptive — Once a process starts on the CPU, it runs until it finishes. Nothing can interrupt it.
Preemptive — The OS can stop a running process mid-run and switch to another (e.g., a higher-priority process arrives, or a timer fires).
operation
Context Switch
The work of pausing one process and resuming another. The OS saves the current process's CPU state (registers, instruction pointer) to its PCB, then loads the next process's saved state. Has a time cost.
visualization
Gantt Chart
A horizontal bar timeline showing which process occupies the CPU at each moment in time. Time flows left to right on the X-axis. Each labeled segment shows one process running. The width of each segment = how long it ran.
P1 runs 0→3, P2 runs 3→5, P3 runs 5→7 → three adjacent colored blocks on the timeline.
algorithm
FCFS — First Come, First Served
Processes run in the order they arrive. The first to arrive runs first to completion, then the next, etc. Non-preemptive and simple.
algorithm
SJN / SRTF — Shortest Job Next
SJN (non-preemptive): at decision time, pick the process with the shortest run time. SRTF (preemptive): if a new process arrives with a shorter remaining time than what's currently running, preempt and run the new one.
algorithm
Priority Scheduling
Each process has a priority number — in this course, bigger = higher priority. The highest-priority ready process always runs. Can be preemptive or non-preemptive.
algorithm
Round Robin (RR)
Each process gets a fixed time slice (quantum) on the CPU. When its slice is up, it goes to the back of the ready queue and the next process gets its slice. Always preemptive. Repeats in a cycle until all finish.
system call
fork()
An OS system call that creates an exact copy (clone) of the current process. No parameters. Returns 0 to the child process and the child's PID to the parent. Both continue running from the same point.
system call
exec()
An OS system call that replaces the current process with a new program. Requires parameters (what program to run). If successful, it never returns — the process IS now the new program.
concurrency
Thread
A lightweight unit of execution inside a process. Multiple threads share the same process memory. Cheaper to create than a new process. Three types: User-Level (runtime manages), Kernel-Level (OS manages), Hybrid.
communication
IPC — Inter-Process Communication
Methods that let separate processes share data: shared files, shared memory, pipes, message queues.
concurrency bug
Race Condition
An unpredictable bug caused by timing: two or more processes access shared data at the same time with at least one writing, and the result depends on who gets there first. Rare, hard to reproduce, dangerous.
Two processes both read counter=5, both add 1, both write 6. Should be 7. One increment was lost.
synchronization
Critical Region / Critical Section
Code that accesses a shared resource and must only be executed by one process at a time. The goal of synchronization is to protect critical regions.
synchronization tool
Semaphore
A special atomic counter used to control access to shared resources. Binary semaphore = 0/1 (mutex). Counting semaphore = integer ≥ 0. Operations must be atomic (uninterruptible) — implemented via hardware: test-and-set() or swap().
synchronization
Mutex — Mutual Exclusion
A binary lock ensuring only one process can be in a critical section at a time. A process "locks" it before entering and "unlocks" it when leaving. Others must wait while it's locked.
concurrency problem
Deadlock
A situation where two or more processes are each waiting for a resource held by the other, so none can ever proceed. Requires all three: Mutual Exclusion + Hold-and-Wait + Circular Wait.
A holds printer lock, needs scanner. B holds scanner lock, needs printer. Both wait forever.
data structure
Circular Buffer
A fixed-size ring-shaped buffer with an IN pointer (where producer writes) and an OUT pointer (where consumer reads). Used for IPC. Needs coordination to avoid reading unwritten data.

Concept Explanations

Every idea explained from scratch, step by step.

01 What the OS Manages — Processes, Memory, Files ▼

An Operating System has three main jobs: managing processes (running programs), managing memory (RAM), and managing files (disk). Processes are the foundation — everything else only matters once a process is loaded and running.

Even on a tiny single-core device (Raspberry Pi Zero, $5), there are typically dozens of processes in some state at any moment. Only one uses the CPU at a time — the rest are either ready and waiting, or blocked waiting for something.

In Linux/Unix you can inspect processes with top or ps -ax, and kill one with kill -9 PID. These tools just make OS system calls — they aren't special or privileged.

02 Process Lifecycle — The Five States ▼

A process doesn't jump straight from "created" to "done" — it moves through states:

NEW
→
READY
→
RUN
⇄
WAIT
→
EXIT
  • New — OS is loading the process from disk into memory. Not yet ready to run.
  • Ready — Process is loaded, waiting in queue for the CPU. Could run, just hasn't been picked yet.
  • Running — Scheduler picked this process; it is actively using the CPU right now.
  • Waiting (Blocked) — Process needs something external: data from disk, a network packet, a timer. It steps off the CPU and waits. When the event fires, it goes back to Ready.
  • Exit — Process finished. OS frees its memory, closes its files, removes its PCB.
📝 The PCB stores everything needed to pause and resume a process: CPU registers, instruction pointer (where it was in the code), open files, memory pointers.
03 Reading & Drawing Gantt Charts ▼

A Gantt chart is a timeline that shows which process is using the CPU at every moment. It's how we visualize scheduling.

How to read one:

  • The horizontal axis is time — moves left to right. Numbers below mark time points (0, 1, 2, 3…).
  • Each colored/labeled block represents one process running. The block starts when the process starts and ends when it stops (finishes or is preempted).
  • The width of a block = how many time units that process ran in that stretch.
  • Blocks next to each other with no gap = CPU was continuously busy. Gaps (sometimes labeled "idle") = no process was running.

Example: P1 runs from time 0→3, P2 from 3→5, P3 from 5→7, P4 from 7→8.

  • Draw a labeled block for each continuous run of a process.
  • Mark the start and end time of each block below the axis.
04 Creating Processes — fork() and exec() ▼

In Unix/Linux, you create new processes with two system calls:

fork() — Clone the current process
1
Process A calls fork()
2
OS creates an almost-identical copy of A (same memory, same open files, same code position). This copy is called the child.
3
fork() returns 0 to the child and child's PID to the parent. This is how they know which one they are.
4
Both parent and child continue running from the line after the fork call.
exec() — Replace current process with a new program
1
Process calls exec() with the name of a new program to run.
2
OS replaces the process's memory/code with the new program.
3
If successful, exec() never returns — the process is now the new program. Same PID, new code.
💡 Common pattern: call fork() first (creates a copy), then call exec() in the child (replaces child with a new program). Parent continues as before. This is how shells run commands.
05 Threads — Lightweight Parallelism ▼

Threads are like mini-processes that live inside a process and share its memory. Creating a thread is much cheaper than creating a new process.

06 IPC — How Processes Talk to Each Other ▼

When processes cooperate (e.g., A reads data → B processes it → C stores it), they need to pass data. This is IPC.

  • Files — A writes to a file, B reads from it. Simple but slow.
  • Shared Memory — Both processes access the same RAM region. Very fast, but requires careful synchronization.
  • Pipes — A one-directional data channel: one end writes, the other reads. Like Unix's | operator.
  • Message Queues — Processes send and receive discrete messages via an OS-managed queue.
⚠️ The key challenge: process B must not read data that process A hasn't finished writing yet. This requires coordination — which is where race conditions and semaphores become important.

RR Round Robin — Always Preemptive (Quantum = 1) ▼

Rule: Every process gets exactly 1 time unit (the quantum) before the next process gets its turn. Goes in a rotating cycle. If a process finishes before its quantum is up, the next one starts immediately.

Important: When a new process arrives at the same time a process finishes its quantum, the new arrival joins the queue first, then the interrupted process goes to the back.

Synchronization & Deadlock

Race conditions, semaphores, and deadlock — explained clearly.

01 Race Conditions — When Timing Goes Wrong ▼

A race condition is an unpredictable bug caused by two processes accessing shared data at the wrong time. The outcome depends on who gets to the data first — like a race.

Classic example — Lost Update:

Shared counter = 5. Two processes both try to add 1.
1
Process A reads counter → gets 5
2
Process B reads counter → also gets 5 (A hasn't written back yet!)
3
Process A computes 5+1=6, writes 6 back.
4
Process B computes 5+1=6, writes 6 back. But it should be 7! One increment was lost.
🚨 Race conditions only happen on data that is both read and written by multiple processes. Read-only shared data is safe — no race condition possible.

The fix: protect the read-modify-write operation so only one process can do it at a time. The area of code doing this is the critical section.

02 Semaphores — The Solution to Race Conditions ▼

A semaphore is a special integer variable with two atomic (uninterruptible) operations: wait (P) — decrement and block if result < 0 — and signal (V) — increment and wake a waiting process.

Why must operations be atomic? If the decrement itself could be interrupted, the semaphore could suffer its own race condition. Hardware provides special instructions (test-and-set(), swap()) to make this one single uninterruptible step.

TypeValuesUse case
Binary (Mutex)0 or 1Protect a critical section — one process at a time
Counting0, 1, 2, 3…Manage N identical resources (e.g., 3 printers)

How a mutex works in practice:

  • Before entering the critical section: call wait() — if semaphore = 1, it becomes 0 and you proceed. If already 0, you block and wait.
  • After leaving the critical section: call signal() — semaphore becomes 1 again and any waiting process is woken up.
🔑 Analogy: a bathroom with one key. Take the key (wait), use the bathroom (critical section), return the key (signal). Nobody can enter while you have the key.
03 Deadlock — When Processes Wait Forever ▼

Deadlock happens when two or more processes are permanently blocked, each waiting for a resource held by the other. It requires all three conditions at once:

  • Mutual Exclusion — only one process can hold a resource at a time (normal for locked resources)
  • Hold-and-Wait — a process holds one resource and waits for another without releasing the first
  • Circular Wait — A waits for B, B waits for A (a cycle)
💀 Example: Process A locks the Printer, then asks for the Scanner. Process B locks the Scanner, then asks for the Printer. Both are stuck waiting forever — deadlock!

Four strategies to handle deadlock:

StrategyWhat happensUsed when
PreventDesign system so at least one of the 3 conditions can never occurSafety-critical: pacemakers, space probes, undersea systems
AvoidDynamically refuse allocations that could lead to deadlock (e.g., Banker's Algorithm)Systems with known resource needs in advance
DetectLet deadlock happen, then detect and recover (kill a process, release resources)General-purpose operating systems
Do NothingIgnore it — reboot when it occurs. Acceptable if very rare.Consumer desktop OSes
💡 Remove any one of the three conditions and deadlock cannot occur. Prevention means structuring the system to violate at least one condition permanently.
04 Circular Buffer — Shared Memory Coordination Problem ▼

A circular buffer is a fixed-size ring of memory used to pass data from a producer process to a consumer process. It has two pointers: IN (where the producer writes next) and OUT (where the consumer reads next).

The race condition problem: The producer moves the IN pointer to a new slot to "claim" it — but it hasn't written the actual data yet. If the consumer checks the buffer at this exact moment, it sees the IN pointer moved and thinks there's new data — but reads garbage or nothing.

The fix: Use a semaphore so the consumer cannot read until the producer has fully written the data. The consumer must wait until the semaphore signals that data is ready.

📝 The circular buffer is a key example of why IPC and shared memory always need synchronization — even a simple pointer move can create a race condition.
