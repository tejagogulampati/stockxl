Your complete exam-ready study guide — definitions, concepts explained simply, and what to expect on the test.

Must-Know Definitions

Memorize these terms — they are the vocabulary of the entire topic.

Process core
A program in execution. When you run a program (e.g., click an app), the OS loads it into memory and starts it — that running instance is a process. Processes are the basic unit of work managed by the OS.
Example: Opening Chrome creates a process. Opening it again creates another separate process.
PID – Process ID identifier
A unique number assigned by the OS to every process. It's how the OS tracks and refers to each process. Think of it like a student ID number.
Example: In Linux, run ps -ax to see the PID of every running process.
PCB – Process Control Block data structure
A data structure the OS maintains for every process. It stores everything the OS needs to know about a process: its PID, name, state, list of child processes, open files, CPU registers, instruction pointer, memory pointers, and accounting info (how long it has run).
Think of the PCB like a student's folder at a school office — it has all the info needed to manage them.
Process States lifecycle
The stages a process moves through during its lifetime:
  • New – just created, being loaded into memory.
  • Ready – loaded and waiting for CPU time.
  • Running – currently executing on the CPU.
  • Waiting/Blocked – paused, waiting for something (I/O, timer, event).
  • Exit – finished and being cleaned up.
Scheduler / Process Scheduler OS component
The part of the OS that decides which process runs, when it runs, and for how long. It uses scheduling algorithms (policies) to make these decisions.
Scheduling Metrics measurement
How we measure how well a scheduler is doing:
  • Wait Time – how long a job sat idle waiting to be scheduled.
  • Turnaround Time – total time from when a job arrived until it completed (includes waiting + running).
  • Throughput – how many jobs complete per unit of time (e.g., jobs/hour).
Turnaround = Wait Time + Run Time
Non-Preemptive Scheduling scheduling type
Once a process is given the CPU, it keeps running until it finishes on its own. The scheduler cannot interrupt it mid-run. Used by FCFS and non-preemptive SJN/Priority.
Preemptive Scheduling scheduling type
The scheduler can interrupt a running process and switch to another one (e.g., when a higher-priority or shorter job arrives, or a timer expires). Used by Round Robin, preemptive Priority, and preemptive SJN.
Context Switch operation
The time and effort required to stop one process and start another. The OS saves the current process's state (registers, instruction pointer) to its PCB, then loads the next process's saved state. This switching has a cost in time.
FCFS – First Come, First Served algorithm
The simplest scheduling policy: processes are run in the order they arrive. Like a queue at a coffee shop — whoever got there first gets served first. Non-preemptive.
SJN / SRTF – Shortest Job Next algorithm
Schedule the process with the shortest run time first. Minimizes average wait time. Can be non-preemptive (pick shortest at start) or preemptive — in which case it's called SRTF (Shortest Remaining Time First): if a new process arrives with a shorter remaining time, it preempts the current one.
Priority Scheduling algorithm
Each process is assigned a priority number (in this class, bigger = higher priority). The scheduler always runs the highest-priority ready process. Can be preemptive or non-preemptive.
Round Robin (RR) algorithm
Each process gets a time slice (quantum) on the CPU, then the next process gets its turn. Always preemptive. Fair — every process gets regular CPU time. Good for interactive systems.
Example: P5 runs for 1 unit → P6 runs 1 unit → P5 runs 1 unit → P7 runs 1 unit → cycling through all ready processes.
Gantt Chart visualization
A timeline diagram used to visualize scheduling. Shows which process runs during which time intervals. The horizontal axis is time; each segment shows which process occupied the CPU.
Fork() system call
An OS system call that creates a copy (clone) of the current process. Takes no parameters. Returns 0 to the child process and the child's PID to the parent. Both processes continue from the same point after the fork.
Exec() system call
An OS system call that replaces the current process with a new program. Has many variations: execv, execve, execl, etc. If successful, it never returns — the process becomes something else.
Thread concurrency
A lightweight unit of execution within a process. Multiple threads share the same process memory and resources but run independently. Three types: User-Level (managed by language runtime, e.g. C library), Kernel-Level (managed by the OS directly), and Hybrid (combination of both).
IPC – Inter-Process Communication coordination
Methods that allow separate processes to exchange data. Common IPC methods: shared files, shared memory, pipes, and message queues.
Race Condition concurrency bug
An unpredictable, timing-dependent bug that occurs when two or more processes access and modify a shared resource at the same time, with the outcome depending on who gets there first ("wins the race"). Results in incorrect, inconsistent behavior.
Example: Two processes both try to update a shared counter at the same time — the final value may be wrong because one overwrote the other's update.
Critical Region (Critical Section) synchronization
A section of code that accesses a shared resource and must not be executed by more than one process at a time. Proper synchronization ensures only one process is in the critical region at any moment.
Semaphore synchronization tool
A synchronization primitive (tool) used to prevent race conditions. Acts as a counter-based lock:
  • Binary semaphore – only 0 or 1 (locked / unlocked). Also called a mutex.
  • Counting semaphore – can hold any non-negative integer, controlling access for multiple slots of a resource.
Critically, semaphore operations must be atomic (single, uninterruptible operation) — otherwise they could themselves suffer a race condition.
Mutex – Mutual Exclusion synchronization
A lock that ensures only one process can use a shared resource at a time. When a process holds the mutex, all others must wait. Implemented via binary semaphore.
Deadlock concurrency problem
A situation where two or more processes are each waiting for the other to release a resource, so none of them can ever proceed. Like two cars blocking a one-lane bridge from each side — neither can move forward or back.
Process A holds Resource 1, needs Resource 2. Process B holds Resource 2, needs Resource 1. Both wait forever.
Deadlock Conditions theory
Three conditions that together can cause a deadlock:
  • Mutual Exclusion – only one process can hold a resource at a time.
  • Hold-and-Wait – a process holding one resource waits for another without releasing the first.
  • Circular Wait – a cycle of processes, each waiting for a resource held by the next.
Circular Buffer data structure
A fixed-size ring-shaped buffer (in memory) used to pass data between processes. Has an IN pointer (where data is written) and an OUT pointer (where data is read). Requires coordination to prevent reading unwritten data or overwriting unread data.

Concept Explanations

Simple explanations of every major idea — built for first-time learners.

01 What is an Operating System's Job?

An OS has three main duties: managing processes, managing memory, and managing files. Processes are the foundation — everything else (memory, files) only matters once a process is loaded and running.

Even a simple single-core device (like a $5 Raspberry Pi Zero) typically has dozens of processes in some state at any moment. Only one actually runs on the CPU at a time — the rest are either ready to run or waiting for something to happen (like a network message or a timer).

💡 On Linux/Unix: use ps -ax to see all processes, or top for a live view. The kill -9 PID command forcibly terminates a process.
02 Process Lifecycle: States a Process Goes Through

A process doesn't just jump straight from "created" to "running" — it moves through several states:

  • 1
    New – OS creates the process, loads it from disk into memory.
  • 2
    Ready – Process is in memory and ready to run, but the CPU is busy with something else. It waits in a queue.
  • 3
    Running – The scheduler picked this process; it is actively executing on the CPU right now.
  • 4
    Waiting/Blocked – Process needs something external (e.g., data from disk, a network packet, a timer to fire). It steps off the CPU and waits. When the event occurs, it goes back to Ready.
  • 5
    Exit – Process finished. OS cleans up its memory and PCB.
📝 The PCB (Process Control Block) stores the process's complete state — registers, instruction pointer, open files — so it can be paused and resumed correctly.
03 Scheduling Algorithms — How the OS Picks Who Runs Next

The scheduler must decide which process gets the CPU next. Here are the main strategies:

FCFS (First Come, First Served) — Non-preemptive

The simplest method: run processes in arrival order. Once a process starts, it runs to completion. Like a queue — no cutting allowed.

⚠️ Downside: a long process arriving first can make many short processes wait a very long time ("convoy effect").

SJN (Shortest Job Next) / SRTF — Can be preemptive or not

Run the process with the shortest estimated run time first. This minimizes average wait time. When preemptive, it becomes SRTF — if a new, shorter process arrives while one is running, the running process is paused and the shorter one takes over.

Priority Scheduling — Can be preemptive or not

Each process has a priority number. In this course: bigger number = higher priority. The highest-priority ready process runs. When preemptive, a newly arrived higher-priority process immediately bumps the current one.

Round Robin (RR) — Always preemptive

Every process gets a fixed time slice (quantum) on the CPU, then the next process in line gets its slice. If a process doesn't finish in its slice, it goes back to the end of the ready queue. This repeats in a circle until all processes are done.

✓ Round Robin is fair — no process starves. It's commonly used in real operating systems for interactive workloads.

04 How to Calculate Wait Time and Turnaround Time

These are the two most important scheduling metrics and very likely to appear as exam calculations.

Turnaround Time
Turnaround Time = Completion Time − Arrival Time
Wait Time
Wait Time = Turnaround Time − Run Time
(or equivalently: Wait Time = Start Time − Arrival Time for non-preemptive)

Example (FCFS): Processes P5 (arrives=0, runs=2), P6 (arrives=0, runs=1), P7 (arrives=1, runs=2), P8 (arrives=2, runs=1). Schedule order: P5 → P6 → P7 → P8.

05 Fork() and Exec() — Creating New Processes

In Unix/Linux, processes are created using two key system calls:

  • 1
    fork() — Creates an exact clone of the current process. The parent and child are nearly identical (same memory, same open files) but have different PIDs. fork() returns 0 to the child and the child's PID to the parent, so they can tell each other apart.
  • 2
    exec() — Replaces the current process image with a new program. After a successful exec, the old code is gone — the process now runs a completely different program. If you call exec after a fork, the child becomes a new program while the parent continues.
📝 system("command") in C is a convenience wrapper — it creates a shell and runs a string. Internally it uses fork + exec. It's powerful but less efficient and potentially unsafe if user input reaches it.
06 Threads — Lighter-Weight Parallelism

Threads are like mini-processes that live inside a process and share its memory. Creating a thread is cheaper than creating a new process. Three models:

  • User-Level threads – Managed by the programming language runtime (e.g., C's clib). OS doesn't know about them. Fast to create, but if one thread blocks, the whole process blocks.
  • Kernel-Level threads – The OS kernel directly manages them. Can run truly in parallel on multi-core systems. More overhead but more power.
  • Hybrid – OS manages kernel threads, each of which can run multiple user threads. Combines benefits of both.
💡 Threads matter most on multi-core systems — kernel-level threads can run simultaneously on different cores, giving true parallelism.
07 Inter-Process Communication (IPC)

When processes cooperate (e.g., a pipeline: A → B → C, where A produces data that B processes that C stores), they need to share data. This is IPC.

IPC Methods:

  • Files – Process A writes to a file; process B reads from it. Simple but slow.
  • Shared Memory – Two processes are given access to the same region of RAM. Very fast, but requires careful coordination.
  • Pipes – A one-directional channel from one process to another (like a physical pipe for data).
  • Message Queues – Processes send and receive discrete messages through a queue managed by the OS.
⚠️ The key challenge with IPC is coordination: process B must not read data that process A hasn't finished writing yet. This is where race conditions and semaphores come in.
08 Race Conditions — When Timing Goes Wrong

A race condition happens when the result of a computation depends on the exact order in which two processes access shared data — and that order is unpredictable.

Simple example: Two processes share a counter variable (value = 5).

  • 1
    Process A reads counter → gets 5
  • 2
    Process B reads counter → also gets 5 (before A writes back)
  • 3
    A adds 1, writes 6 back
  • 4
    B adds 1, writes 6 back (should be 7!)

The increment was lost! This is a race condition — rare, hard to reproduce, and very hard to debug.

🚨 Race conditions happen when shared data is both read and written by multiple processes without synchronization. Read-only shared data is safe.
09 Semaphores — Solving Race Conditions

A semaphore is a special counter variable used to control access to a shared resource. The key rule: semaphore operations must be atomic — they cannot be split across processes. They're implemented using hardware instructions like test-and-set() or swap().

  • Binary Semaphore (Mutex) – Value is 0 (locked) or 1 (unlocked). Before entering a critical section, a process does a "wait" (decrement). After leaving, it does a "signal" (increment). Only one process can be inside at a time.
  • Counting Semaphore – Value can be any non-negative integer. Useful when you have N instances of a resource (e.g., 3 printers). Processes decrement to claim one and increment to release it.
✓ Think of a semaphore like a bathroom key at a gas station: you take the key (decrement), use the bathroom (critical section), return the key (increment). Nobody else can enter while you have the key.
10 Deadlock — When Processes Wait for Each Other Forever

Deadlock is a situation where two or more processes are stuck waiting for each other, and none can ever proceed.

Classic example: Process A holds a lock on the printer and needs the scanner. Process B holds a lock on the scanner and needs the printer. Both wait forever.

For deadlock to occur, all three conditions must be present simultaneously:

  • Mutual Exclusion – resources can't be shared simultaneously
  • Hold-and-Wait – a process holds a resource while waiting for another
  • Circular Wait – a cycle of dependency (A waits for B, B waits for A)

What can be done about deadlock?

StrategyWhat it meansWhen used
PreventDesign the system so one of the 3 conditions can never occurSafety-critical systems (pacemakers, space probes)
AvoidUse algorithms that dynamically refuse resource allocations that could lead to deadlockComplex real-time systems
DetectLet it happen, but detect it and recover (kill a process, release resources)General-purpose OS
Do NothingAssume it's rare enough not to matterDesktop OSes (reboot fixes it)
11 Circular Buffer — Shared Data Structure Between Processes

A circular buffer is a ring-shaped fixed-size array used when one process produces data and another consumes it. Think of a conveyor belt at a sushi restaurant — new plates are added at one end and taken from the other.

  • An IN pointer marks where new data is written (producer advances it after writing).
  • An OUT pointer marks where data is read (consumer advances it after reading).
  • The buffer is "circular" — when a pointer reaches the end, it wraps around to the beginning.

The synchronization problem: if the producer moves the IN pointer before writing the actual data, the consumer might read garbage. Coordination (semaphores) is needed to prevent this.

Exam Tips & Likely Questions

What your professor is most likely to test — organized by topic.

⏱ Scheduling Algorithms
Very Likely
Trace through a scheduling algorithm by hand. Given a table of PIDs, arrival times, run times, and priorities — draw the Gantt chart and compute wait/turnaround times for each algorithm (FCFS, SJN, Priority, RR). Practice this until it's automatic.
Very Likely
Know the formulas cold: Turnaround = Completion − Arrival and Wait = Turnaround − Run Time. The exam will almost certainly ask you to calculate these.
Very Likely
Preemptive vs. Non-Preemptive — Be able to explain the difference and identify which algorithms are which. Know that Round Robin is always preemptive, and FCFS is always non-preemptive.
Likely
Compare algorithms: Which gives the best average wait time? (SJN/SRTF). Which is fairest? (Round Robin). Which is simplest? (FCFS). Be ready for "what are the tradeoffs" questions.
Likely
Context switch — Know that switching between processes has a cost (time). This is especially relevant for RR: if the time quantum is too small, you spend more time on context switches than actual work.
⚙️ Processes & PCB
Very Likely
List and explain process states (New → Ready → Running → Waiting → Exit). Be able to describe what causes a transition between states (e.g., what moves a process from Running to Waiting?).
Very Likely
What is a PCB and what does it contain? Be able to list at least 5 fields: PID, name, child process list, accounting info, open files, CPU registers/instruction pointer, memory pointers.
Likely
fork() vs exec() — Know the difference. fork() copies the process; exec() replaces it. Know that fork() returns 0 to the child and the child's PID to the parent.
🔒 Synchronization & Race Conditions
Very Likely
Define race condition and give an example. The key elements: shared resource, multiple processes reading AND writing, unpredictable outcome based on timing.
Very Likely
What is a semaphore? Know the difference between binary (mutex) and counting semaphores. Know that semaphore operations must be atomic (single, uninterruptible) — this is why normal variables can't be used instead.
Very Likely
Critical region / critical section — Define it. Explain why only one process should be in the critical region at a time, and how semaphores enforce this.
Likely
Circular buffer race condition — Describe the specific problem: IN pointer moves before data is written, so the consumer reads garbage. The solution is coordination using semaphores.
💀 Deadlock
Very Likely
List and define the three deadlock conditions: Mutual Exclusion, Hold-and-Wait, Circular Wait. Understand that all three must be present for deadlock to occur — remove any one and deadlock cannot happen.
Very Likely
Four deadlock handling strategies: Prevent, Avoid, Detect, Do Nothing. Know when each is appropriate (e.g., prevention for safety-critical systems like pacemakers).
Likely
Give a concrete deadlock example: two processes, two resources, each holding one and waiting for the other. Be able to draw the cycle and identify which condition each requirement maps to.
🧵 Threads & IPC
Likely
Three thread models: User-Level, Kernel-Level, Hybrid. Know the key difference — who manages the threads (language runtime vs. OS kernel) and how each interacts with multi-core CPUs.
Likely
IPC methods — Be able to list at least three (files, shared memory, pipes, message queues) and briefly explain each. Know the general coordination challenge that comes with shared memory IPC.
Definition
Process vs. Thread — A process is a full program in execution with its own memory space. A thread is a unit of execution within a process, sharing the process's memory. Threads are lighter-weight than processes.
